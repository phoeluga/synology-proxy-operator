# Development guide

This guide covers building, testing, linting, and working on the operator locally.

For a full local cluster walkthrough with minikube and the VSCode debugger, see [local-testing.md](local-testing.md).

---

## Prerequisites

| Tool | Version | Install |
|---|---|---|
| Go | ≥ 1.22 | `brew install go` |
| Docker | any | Docker Desktop |
| minikube | any | `brew install minikube` |
| kubectl | any | `brew install kubectl` |
| Helm | ≥ 3 | `brew install helm` |

Install local toolchain (controller-gen, envtest):

```bash
make controller-gen
go mod tidy
```

---

## Build

```bash
make build          # compile binary → bin/manager
make fmt            # gofmt
make vet            # go vet
make lint           # golangci-lint (requires golangci-lint installed)
```

---

## Test

```bash
make test                                              # full suite with envtest, outputs coverage.out
go test ./internal/controller/... -run TestFoo -v     # single test by name
go test ./... -coverprofile=coverage.out && \
  go tool cover -html=coverage.out                    # view coverage in browser
```

Tests use [envtest](https://book.kubebuilder.io/reference/envtest) — a real Kubernetes API server and etcd run in-process. No cluster needed.

## CRD generation

After editing any type in `api/v1alpha1/`:

```bash
make generate    # regenerates zz_generated.deepcopy.go
make manifests   # regenerates config/crd/bases/*.yaml and config/rbac/role.yaml
```

Then copy the updated CRD to the Helm chart:

```bash
cp config/crd/bases/*.yaml helm/synology-proxy-operator/crds/
```

The `.claude/settings.json` hook will remind you to run these commands whenever you edit a file under `api/v1alpha1/`.

## Container image

```bash
make image-build                                             # build with Docker
make image-build CONTAINER_TOOL=podman IMG=operator:dev     # build with Podman
make image-buildx IMG=ghcr.io/phoeluga/synology-proxy-operator:dev  # multi-arch (amd64+arm64)
make image-load                                              # load into minikube or kind
```

## Local development cluster

```bash
make minikube-start    # starts minikube (4 CPUs, 6GB RAM, ingress addon, docker driver)
make dev-setup         # installs CRDs + RBAC + ArgoCD + test fixtures
make dev-run           # runs the operator locally, reading .env.local for DSM credentials
make minikube-stop     # shuts down the cluster
make minikube-delete   # tears it down completely
```

Create `.env.local` from the example before running `dev-run`:

```bash
cp .env.local.example .env.local
# edit .env.local with your Synology DSM credentials
```

## Linting

The project uses [golangci-lint](https://golangci-lint.run/) with configuration in `.golangci.yml`.

```bash
make lint
```

The enabled linters are defined in `.golangci.yml`. The CI pipeline runs the same command with the same version — add or remove linters there, not just locally.

---

## IDE setup (VSCode)

`.vscode/launch.json` provides a "Run operator (minikube)" debug configuration. After `make dev-setup`:

1. Open the project in VSCode
2. Press **F5** to start the operator with the debugger attached
3. Set breakpoints in any controller or client file

`.vscode/tasks.json` provides build and test tasks accessible from the command palette.

## Code generation note

`api/v1alpha1/zz_generated.deepcopy.go` is auto-generated by `controller-gen`. Never edit it manually — changes will be overwritten on the next `make generate`.

The `config/crd/bases/` and `config/rbac/` directories are also generated. Do not edit them directly; change the kubebuilder markers in the source files and re-run `make manifests`.
